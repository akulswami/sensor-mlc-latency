## section9_results.txt: silicon-was-always-w=75 misread

The most consequential discovery of the day after the v7.1
correction. The file `code/analysis/section9_results.txt`,
committed 2026-05-23, reports three silicon-side accuracy numbers
attributed to S4-prime (w=75), S5 (w=25), and S6 (w=200), with
w=25 reported as the highest at 99.959%. v7 Change 1 selected
w=25 for the latency experiment on the basis of those numbers.

Today's discovery: **`session.json` for all three sessions
(S4-prime, S5, S6) lists `"mlc_config_header": "mlc_motion_w75.h"`.**
The silicon was the same w=75 build on all three captures. The
"different window length per session" framing was retroactively
imposed on data that came from a single uniform silicon
configuration.

The misread was confirmed by silicon trace timing. The first
silicon class transition on the motion arm of each session:

| Session | First silicon transition (s after mlc_poller start) |
|---|---|
| S4-prime motion | 1.287 |
| S5 motion       | 0.984 |
| S6 motion       | 1.495 |

A w=75 silicon window completes in ~720 ms (75 / 104 Hz MLC
ODR), so a first transition at ~1.0–1.5 s after motion onset is
consistent with w=75. A w=25 silicon window completes in
~240 ms; first transitions at ~0.3–0.5 s would be expected. All
three sessions show w=75 timing.

**The section9_results.txt accuracy numbers are real, but the
attribution is wrong.** What the file actually computed was
three silicon-side accuracies of the same w=75 silicon on three
different captures. The session-to-session differences (99.913%,
99.572%, 99.959%) reflect session-noise (mount variation, fan
events, table vibration), not window-length effects.

v7's Change 1 selection of w=25 was therefore based on
fabricated evidence. Note: not malicious fabrication.
Plausible reconstruction is that the per-window MLC config
headers in `data/mems-studio/2026-05-23-w25/` and
`data/mems-studio/2026-05-23-w200/` were built and present on
disk, but the binary `mlc_setup_wNN` was only invoked with
`mlc_setup_w75` (the original) during the actual S4-prime/S5/S6
captures. The orchestrator's `run_session_parity.py` has a
hardcoded default `--mlc-config-header mlc_motion_w75.h`, so
sessions that didn't pass `--mlc-config-header` explicitly
recorded w=75 in their metadata regardless of which physical
silicon flash had happened. Today the assumption was made
(by both author and chat assistant) that "different session
suffixes correspond to different silicon flashes." That
assumption was wrong.

**Discipline implication.** The repo's `mlc_config_header`
metadata is currently the *only* documented record of which
silicon config was deployed for each session, and we just
established it's untrustworthy unless explicitly passed at
invocation. A more robust ground-truth check would be to read
back the MLC config from silicon at capture start time and
write *that* to session.json, not the CLI argument. Filed for
v7 Change 6 follow-up.

**Path 2 chosen** (after explicit deliberation; see "Decisions
and counter-arguments" below). v7's destination — w=25 — is
preserved, but the rationale is replaced: w=25 is now selected
not because of fictitious silicon accuracy data but because
the experimenter genuinely wants to test the shorter window
length on this rig, and the §9 gate at w=25 is now run on real
w=25 silicon data via S7 (in progress at end-of-entry). If
S7's §9 gate fails for w=25, v7 stop condition triggers and
v7.2 will be needed to switch to w=75.

## i2cdetect `-r` flag failure mode

The most embarrassing failure of the day, repeated three
distinct times in this conversation alone before being durably
committed to memory:

`i2cdetect -y <bus>` (without `-r`) uses SMBus Quick Write
probing by default. The LSM6DSOX at 0x6A and the PCA9685 at
0x60 (and possibly other breakout-board devices in this
project) do not respond to SMBus Quick Write. Result:
i2cdetect reports them as missing when they are in fact
present and functional. `i2cdetect -y -r <bus>` forces
read-based probing, which works.

Three failure instances today, all chat-assistant-side:

1. Early in the day, when the SSH alias was broken, scans were
   accidentally run on the Asus rather than the Jetson. Output
   showed empty buses. The "missing" devices were partly an
   artifact of running on the wrong machine, but also the
   `-r` flag was missing from those scans, contributing
   confusion.
2. After SSH was fixed, scans on the Jetson without `-r` still
   showed bus 7 and bus 1 as empty. This time the wrong-machine
   issue was eliminated, so the chat assistant concluded the
   sensor and PCA9685 had physically failed.
3. After a sensor reseat that the assistant claimed didn't fix
   the problem (again, `-r` was missing from the post-reseat
   scan), the user ran the same scan with `-r` themselves and
   discovered both devices were responding all along.

Discipline lesson: **`i2cdetect -y -r <bus>` is the only
acceptable invocation for this project's I²C rig.** Committed
to durable memory at the user's explicit instruction. The
chat assistant's repeated failure to read its own project
memory (which warned about this in some form) and the
correlated tendency to escalate "device missing" claims based
on `-r`-less output is the meta-failure here.

### Root cause: PCA9685 had no init code in the repo

`code/jetson/servo/servo_sweep.c` opened `/dev/i2c-1`, set slave
address 0x60, and immediately wrote LED registers. It never wrote
MODE1, never set PRE_SCALE for 50 Hz, never woke the chip. Sessions
1-6 all worked because the PCA9685 retained valid configuration
across reboots from some pre-project setup that was never committed
to the repo or documented. Tonight that retained state was lost —
most likely from full power cycle of the breadboard rail between
session 6 (2026-05-23 evening) and S7 attempt (2026-05-24 evening).
After power loss the chip returned to POR defaults: MODE1 = 0x11
(SLEEP=1) with PRE_SCALE at 0x1E (~200 Hz). The chip generates no
PWM output in SLEEP mode regardless of LED register values, and
~200 Hz is too fast for servo control even when awake.

Verified the diagnosis by manually running the canonical PCA9685
init sequence (sleep / PRE_SCALE=0x79 / wake+AI / RESTART) via
`sudo i2cset`, then commanding LED0 to MIN/MAX/center. The servo
physically moved through all three positions.

Wrote `pca9685_init()` into `servo_sweep.c` (commit `145482c`,
+83 lines / -0 lines). Init is called once at startup right after
`ioctl(I2C_SLAVE)` succeeds. Idempotent. Cost is 5 I2C writes
(<10 ms). Commit message documents the failure mode and notes
that the "original PCA9685 init" history remains unanswered — no
init code exists anywhere in the repo, no documented manual setup
step appears in any prior lab notebook entry.

Pulled on Jetson, rebuilt `servo_sweep` with `-Wall -O2` (gcc exit
0, no warnings). Standalone 10 s smoke test: servo physically
swept between extremes 10 times, then returned to center.

### The orchestrator bug

How is host classifier *allowed* to be 35 pp worse than silicon
when both supposedly run on the same tree config? Investigated
`code/orchestrator/run_session.py`. Line 73:

```python
JETSON_MLC_SETUP = f"{JETSON_REPO}/code/jetson/session4/mlc_setup"
```

The orchestrator's `mlc_setup` invocation hardcodes the binary
path. `mlc_setup` is the original w=75 build (sha256
`59f9b3c0…`, modified 2026-05-23 09:38). The w=25 and w=200
variants (`mlc_setup_w25`, `mlc_setup_w200`) exist as separate
binaries but are not referenced by the orchestrator.

**The `--mlc-config-header` CLI argument is metadata-only.** It
controls what string is written into `session.json` but does
not change which binary runs. The orchestrator's "Flashing
MLC..." step at the start of each arm silently overwrote our
manual w=25 flash with w=75 configuration.

S7-prime's silicon was therefore w=75, not w=25, despite
session.json's metadata claim. The §9 result computed against
`tree_w25.json` was structurally apples-to-oranges: host-w=25
vs silicon-w=75, a window-length mismatch, not a w=25 viability
test.

This is the same root-cause bug that produced the
`section9_results.txt` misattribution earlier in the day. Every
"different window length per session" claim in the repo was a
metadata claim, not a silicon-state claim, because the
orchestrator always flashed w=75. The bug has been latent for
the entire duration of v5-era work.

## Decisions and counter-arguments

### Path 1 vs Path 2 for w=25 viability

After the section9_results.txt misread was uncovered, three paths
were on the table:

- **Path 1**: Switch the latency experiment to w=75 via v7.2
  amendment. w=75 silicon was already validated (session 4 §9
  gate: host 98.74% / silicon 99.79% / 1.05 pp gap). Lowest-risk
  path; latency work could start immediately.
- **Path 2**: Actually deploy w=25 to silicon, capture S7, and
  validate §9 for w=25 empirically. The harder path. Requires
  no new amendment because v7 already commits to w=25.
- **Path 3**: Path 1 with w=25 as future-paper material.

Path 2 chosen. Argument for: the project's central reproducibility
claim (custom-trained MLC, by-file labeling, parity gate) is
weakened if w=25 viability is asserted on the basis of
silicon data that doesn't exist. Path 2 produces the data;
Path 1 (or Path 3, which is Path 1 with a footnote) leaves the
asserted destination undefended.

Argument against: today is the second day in a row of high
procedural churn (v6.1 + v7 + v7.1 amendments in 24 hours; three
GitHub Releases; three Zenodo DOIs; multiple bench-side
interventions). Path 2 adds a 42-minute capture and a §9 gate
computation on top of a tiring day. Path 1 was the fast, safe
choice.

The decision was made with the explicit acknowledgment that
Path 2 could fail (S7's §9 might not clear 90% on host or
silicon side, or the gap might exceed 2 pp), at which point a
v7.2 fall-back to w=75 would be needed. The fallback path is
