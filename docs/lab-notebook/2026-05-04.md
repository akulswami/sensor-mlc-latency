# 2026-05-04: First clean wire-level latency comparison with piezo ground truth

## Setup
- Piezo (27mm disc, taped to sensor breakout) on Saleae A0 as ground truth
- Saleae D0 = sensor INT (Pin 15), D1 = decision GPIO (Pin 11)
- Trigger: rising A0 above 1.0V, hysteresis at 0.5V
- 12-15 hand-taps per phase, ~1 sec apart
- Reset-hardened latency_test.c and host_pipeline.c (SW_RESET, retry on EREMOTEIO,
  WHO_AM_I sanity check, drain latched ints before reset)

## Solenoid actuator path abandoned
Spent ~5 hours debugging Pin 29 (PQ.05) GPIO drive into MOSFET gate.
Pin 29 outputs 3.34V into open circuit but only 1.0V into 100K+gate load.
Root cause: TXB-style level shifter on Orin Nano dev kit pin has limited
drive current; 1KΩ pulldown didn't help. First IRLB8721 also damaged during
debugging (gate-drain shorted, replaced).

Decision: drop solenoid for the paper. Hand-tap + piezo ground truth is
publishable methodology -- piezo fires at moment of physical impact, so
each individual measurement is precise. Mechanical actuator can be
"future work" for n=2000 automation.

## Data
- Phase 1 (on-sensor pipeline / latency_test.c with hardware single-tap detector):
  n=20 piezo->D1 events
  median=12.54 ms, range 9.18-16.28 ms
- Phase 2 (on-host pipeline / host_pipeline.c with sliding-window p2p classifier):
  n=14 piezo->D1 events  
  median=6.49 ms, range 4.03-8.70 ms

## Surprising finding
On-host pipeline is ~2x faster than on-sensor at median, contrary to
H1 hypothesis. Likely cause: LSM6DSOX hardware tap detector requires a
"quiet" window after the shock event before INT1 fires. Configured at
REG_INT_DUR2 = 0x06 = 6 ODR samples = ~14.4 ms quiet-duration, which
matches the observed 12.5 ms median. Algorithmically-required latency,
not I/O bottleneck.

The on-host sliding-window classifier doesn't require quiet-window;
fires as soon as the rolling 50ms window contains a peak crossing 1g.

## NOT YET ANALYSIS-READY
- n is bring-up only (20+14 events vs pre-reg's 2000 per condition)
- Classifier accuracy parity not yet measured (pre-reg §9 gate)
- on-sensor uses hardware tap-detect as MLC stand-in; actual MLC programming pending
- on-sensor INT_DUR2 is configurable; current 0x06 is one option, not optimal

## Methodology contamination found earlier
Initial Phase 1 retake gave median 1.27ms, but D0 fired 12,645 times in
30s -- far higher than 32 piezo events. Cause: latency_test.c didn't
SW_RESET the chip, so DRDY config from prior host_pipeline run persisted.
INT1 fired on every accel sample, latency analysis matched piezo events
to coincidental D1 edges. After adding SW_RESET hardening to latency_test.c,
clean Phase 1 produced 20 D0 events for 20 actual taps as expected.
