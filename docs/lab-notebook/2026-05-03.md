# 2026-05-03

## B1: INT pulse capture verified
Configured LSM6DSOX hardware single-tap detector via tap_int_test.py.
Threshold 500 mg per axis, ODR 416 Hz, single-tap mode, INT routed to I1 pin.

Saleae Logic Pro 8 connected to sensor I1 pin (D0) and GND. Captured at
25 MS/s, trigger on rising edge, 5 second buffer, 12.75 s window.

Tapped breadboard 4-5 times. 4 TAP_DETECTED events reported by host
(all X-axis, mixed sign as expected for taps in both directions).
4-5 corresponding rising edges visible on Saleae D0. Pulse widths
narrow (well under 50 ms host polling interval). Edges clean.

Wire-level measurement chain validated end-to-end on the sensor-INT side.

## B3: GPIO output toggling verified end-to-end via libgpiod from C
Wrote gpio_squarewave.c. Targets gpiochip0 line 112 (PR.04 = pin 11).
Compiled with: gcc -O2 -Wall -o gpio_squarewave gpio_squarewave.c -lgpiod
Drives line at nominal 100 Hz for 5 seconds.

Saleae D1 captured the resulting square wave on Pin 11.
Measured period: 10.121 ms (target 10.000 ms; +1.2% systematic offset).
Duty cycle 50%. Edges sharp (sub-40-ns transitions at 25 MS/s).
Uniform period across the capture - offset is systematic, not jitter.

Cause of offset: userspace nanosleep oversleeps by ~60 us per call due to
syscall + scheduler granularity. For the actual host pipeline (event-driven
read-on-INT, no sleeps), this overhead does not apply. Noted for the
methods section.

GPIO output chain (C -> libgpiod -> kernel -> SoC -> pin -> Saleae) verified.

## B4: First wire-level latency measurements (on-sensor pipeline, bringup)
latency_test.c: I2C-configures hardware single-tap detector, blocks on
INT (Pin 15 = gpiochip0 line 85) via libgpiod rising-edge events, reads
TAP_SRC, toggles decision GPIO (Pin 11 = gpiochip0 line 112).

Tapped breadboard ~12 times. Tap detector fired 27 events (rebound +
finger-lift secondary triggers; will need debounce or double-tap mode
in actual experiment).

Saleae captured 15s @ 25 MS/s, both D0 (INT) and D1 (decision).
27/27 D0 risings matched a D1 rising before the next D0 rising. Clean.

Wire-level latency, on-sensor pipeline, no stress, no MLC yet:
  n=27, min=221us, p25=242us, median=652us, mean=720us,
  p75=1049us, p95=1550us, max=1838us.

Distribution is bimodal: tight cluster at 221-243us (n=8), gap, then
910-1090us cluster (n=7), tail to 1838us. Likely cause: CPU scheduling
+ frequency scaling + I2C read overhead. Not yet stress-tested or with
jetson_clocks/MAXN locked.

Bringup data only; not part of pre-registered experiment.

CSV exported, processed by code/analysis/extract_latency.py.
Raw and processed files in data/raw/ and data/processed/ on the laptop.
