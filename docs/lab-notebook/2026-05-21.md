### Subtask C2.1: AN5259 read, byte-format dead end

Before writing any tree-decoding code, attempted to fetch the
byte-level decision-tree node format from AN5259. **The PDF does not
specify it.** Section 1.4 describes the tree conceptually (Figure 19;
"each node contains a condition where a feature is evaluated against
a threshold") but defers the actual encoding to "the software tool
provided by STMicroelectronics".

Confirmed via web search that no public byte-level decoder for the
LSM6DSOX MLC tree exists. ST community thread (2024) has an ST
employee explicitly saying: "since are quite specific procedures we
don't support writing those configuration by hand but only via the
cited tools." Reverse-engineering the `.ucf` byte stream is not a
sanctioned path.

**Decision: do not reverse-engineer the byte format.** Use MEMS
Studio's JSON export instead. MEMS Studio 2.0+ emits JSON natively
(per ST blog post Apr 2025) for the express purpose of letting
developers read/edit/port MLC configs.

This forced a small redesign of the C2 work: the host port can't
load a `.ucf` directly. It loads a stable `tree.json` we control,
produced by a Python converter (`mlc_json_to_parity.py`) that reads
MEMS Studio's JSON.

### Subtask C2.3: replay_parity.c (offline harness)

Wrote a standalone offline replay harness that reads a MEMS Studio
HSDataLog CSV, runs it through a configurable feature pipeline +
decision tree, and emits per-window decisions to stdout in a stable
CSV schema. No hardware; pure replay. Tree loaded at runtime from
JSON, not compiled in — one binary serves every training iteration.

Hand-rolled JSON subset parser (no external deps). Implements the
decimation policy from AN5259 §1.1 (sensor samples → MLC samples via
integer ratio).

Wrote `test_replay_parity.sh` as a unit test: generates a 300-sample
synthetic CSV with known still→motion→still pattern, hand-written
3-node toy tree, asserts 12/12 windows classify correctly.

**First test draft FAILED** because the synthetic-test threshold I
picked (0.05 g p2p) was too high for the 25-sample / 120 ms window
to reliably capture a 1 Hz sinusoid: windows landed on near-flat
portions of the cycle. Lowered threshold to 0.005 g (still windows
yield p2p=0 exactly, motion windows hit p2p ≥ 0.03). PASS 12/12.

**The phase quirk is real and worth carrying forward:** at MLC
window cadence, low-frequency motion stimuli produce wildly variable
per-window features depending on phase alignment with the window
boundary. This will affect tree training and possibly future stimulus
design.

### Subtask C2.4: refactor — extract parity_core

Initial `replay_parity.c` and `host_pipeline_parity.c` both
contained their own copy of feature pipeline, filter math, JSON
parsing, decimation, and tree-walk. Extracted to `parity_core.{c,h}`.
Single source of truth.

Public entry point is `pc_step(cfg, ax, ay, az, &cls, feat_out)`:
caller feeds every sensor sample; pc_step internally handles
decimation, filters, windowing, feature computation, and tree walk;
returns true only when a window boundary at MLC cadence is reached,
with the class code written to `cls`. This is the natural
sample-stream interface for both the CSV reader (read line, call
pc_step) and the DRDY interrupt handler (wake on edge, read I2C,
call pc_step).

Verified the refactor by re-running the unit test: output
byte-identical to the pre-refactor run across all 12 windows, every
var_norm and p2p_norm matching to 6 decimal places. Strongest
possible behavior-preservation signal.

Slim-down: replay_parity.c 770 → 221 lines (71%);
host_pipeline_parity.c 470 → 315 lines.

Commit: 467e1f5 (refactor), d6bd301 (warning fix on Jetson build).

### Subtask C2.6: compare_decisions.py

Wrote `code/analysis/compare_decisions.py`. Diffs two CSV files in
the replay_parity output schema, reports per-window class agreement.

Design choices worth noting:

- **Strict equal-length contract.** Row-count mismatches are
  parity failures, not occasions to truncate. A mismatch indicates
  the two pipelines disagreed about whether a window boundary
  occurred at all (decimation phase mismatch, wrong window length).
- **Compare only the `class` column.** AN5259 §1.2 specifies that
  MLC filter coefficients are in half-precision float; AN5259 does
  NOT specify the precision of feature computation itself. Either
  way, the host's float32 filter pipeline cannot bit-match MLC's
  half-precision filter, so feature values will differ between
  host and silicon by construction. Only the threshold-crossing
  decision has to agree.
- **First-divergence reporting includes both rows' feature values**
  so the operator can see whether a disagreement is a near-miss
  (classifier on either side of a threshold) or a gross feature
  mismatch (suggesting filter sign error or NORM vs NORM_2 confusion).

Tests 4/4 PASS: identical, single-mismatch, length-mismatch,
malformed-input. Commit: c8dd27c.

### Session 1 analysis findings

Ran `replay_parity` against session 1's still and motion CSVs using
a toy threshold tree (p2p > 0.005 g ⇒ motion). Not the trained
classifier (we don't have one); the run was to validate the harness
against real noisy data and to characterize the feature distributions.

**Run setup:**
- window_length = 75 samples
- sensor_odr = 208 Hz, mlc_odr = 104 Hz, decim = 2
- HP filter bypassed (b1=1, b2=0, a2=0)
- ~1200 sec per class → 1697 windows per class

**Feature distributions (post-filter):**

| Metric | Still (class=0) | Motion |
|---|---|---|
| var_norm | 2.4e-7 to 9.4e-7 | 3.3e-6 to 3.7e-3 |
| p2p_norm | 1.9e-3 to 4.9e-3 | 1.6e-2 to 4.5e-1 |

**Min separation:** variance 3.5× clean gap, p2p 3.3× clean gap.
A trained tree should find thresholds with zero leakage on either
feature.

**Observation 1: variance is the cleaner feature than p2p.**
Within the still class, variance ceiling (9.4e-7) is well-defined.
p2p has 7 outlier windows above its 4.9e-3 ceiling, reaching up to
8.1e-3. p2p reflects only the two extrema of a window (max minus
min), so a single noisy sample dominates the result. Variance
averages squared deviations over all 75 samples and is robust to
single outliers. **Worth recommending in the paper's discussion
section** that future replications could drop p2p, use variance
alone, run a depth-1 tree.

(Do not change the spec mid-run — the pre-reg commits to both
features. The observation is for the discussion section, not a
methodology change.)

**Observation 2: bench warmup effect.** The 7 p2p outliers in the
still class are not uniformly distributed across the 1200 s
recording. Six of seven fall in the first 60 seconds (t=7.8, 11.3,
24.7, 54.4, 55.1, 58.7 s); the seventh is at t=769 s.

Rate-normalized: first 60 s contains ~85 windows, of which 6 are
outliers → 7.1%. Remaining ~1140 s contains ~1612 windows, of
which 1 is an outlier → 0.06%. That's a ~100× higher outlier rate
during the first minute. Not a sample-size artifact.

Likely cause: bench / mount / electronics settling after session
startup. Recommend adding a session warmup discard to the spec —
drop the first N seconds of still data per session, where N = 60
looks defensible from this single observation. Do not commit to
this until sessions 2 and 3 confirm the pattern (n=1 is not enough).

**Observation 3: motion variance dynamic range is ~1000×.**
3.3e-6 to 3.7e-3 across motion windows, three orders of magnitude.
This is the phase-quirk effect from the unit test, but on real
servo motion at the bench. Adjacent windows can have wildly
different captured variance depending on where the swing happens
relative to window boundaries. Doesn't threaten the classifier
(min motion variance is still 3.5× the still ceiling) but is worth
documenting because reviewers may ask why the variance histogram
is so broad.

**Observation 4: 1697 = 1697 windows across both files.** The
orchestrator's duration control is working to within ~3 ms.

### What went wrong and lessons

The mistake was operating on the handoff's framing of session
spacing as "≥24 h" without re-reading the actual pre-registration
spec. The spec is permissive on time and strict on re-setup;
I had it backwards.

What I should have done at the start of tonight's session-2 work:
re-read `training-data-spec.md §Sessions and train/test split`,
not the handoff. The handoff is a derived artifact, not the
binding contract. Going forward, before any methodology-relevant
action: read the spec section that governs it.

What worked: noticing the X stddev was suspiciously identical to
session 1 (0.000761 vs 0.000766) within the post-session check.
The variance-ratio gate alone wouldn't have caught this; the
re-mount-verification math (3.2° tilt) was what made the
spec-conformance question explicit. Empirical checks beat
checklist checks when the checklist itself is incomplete.
