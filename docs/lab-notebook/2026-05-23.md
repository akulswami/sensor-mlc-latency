#### The `pkill -f` fix in run_session_parity

While writing run_session_parity, I initially used `pgrep -f mlc_poller`
+ `sudo pkill -f mlc_poller` for the post-class straggler cleanup. The
`-f` flag matches the full command line including arguments. Any
process whose argv contains the substring "mlc_poller" — including
the orchestrator's own SSH wrapper running that very `pgrep` — would
match. Worst-case scheduling order could kill the SSH session
mid-run or kill an unrelated process that happens to share the
substring.

Fix: switched to `pgrep -x mlc_poller` + `sudo pkill mlc_poller`,
which matches the exact process basename only. The `.bak` file
preserved the pre-fix version during editing and was deleted before
committing.

**Caveat:** the fix was made ~28 minutes before still capture
started, so the entire session 4 capture ran on the corrected
version. But the straggler-kill path only fires if `pgrep` finds a
leftover process after the duration timer expires, and no
orchestrator log was saved from this run. Whether the straggler
branch fired or not is unknown. The fix is conceptually right but
its session-4 exercise status is undetermined.

### The 11 still-arm transitions are interesting

207 polls at class=4 on the still arm correspond to ~4.14 s of
cumulative silicon-reported "motion" during a recording where the
rig was ground-truth still. 11 transitions ≈ 5 complete blips (each
enter+exit = 2 edges) plus one dangling edge — consistent with five
discrete motion-like events plus one event clipped by the recording
window.

What's notable is **how large the p2p excursion has to be to
trigger class=4**. The trained threshold is 0.049316 g; session 1's
still-class p2p ceiling was 4.9e-3 g (three orders of magnitude
below threshold). For silicon to flip class, the window's filtered
peak-to-peak has to clear ~10× the typical still-noise ceiling —
not a borderline-classifier event. Three hypotheses:

1. **Real low-amplitude environmental events.** Bench settling,
   building HVAC, foot traffic, electrical-equipment coupling.
2. **Silicon-only half-precision filter artifact.** The MLC's
   half-precision IIR1 filter could produce spurious large
   transient outputs that the host's float32 implementation
   smooths.
3. **Rig wear over the 20-minute still recording.** Less likely
   in 20 min; included for completeness.

**The §9 parity gate distinguishes between these directly.** If the
host classifier on the same accel.csv stream produces 11 transitions
at the same timestamps, the events are real signal (option 1) and
host-silicon parity holds for them. If the host shows zero
transitions and silicon shows 11, the events are silicon-only
(option 2) and the gate fails on those windows. Either outcome
informs the paper: the former says both pipelines see the same
world; the latter is a concrete demonstration of host/silicon
disagreement at low-amplitude transients, which is exactly the
kind of finding the paper's central question (on-sensor vs on-host
inference) can discuss.

Will not investigate further today. The data to answer this is
captured; the analysis is stage 2/3 of the parity gate post-process.

#### §9 reading note

The pre-registration's §9 specifies the gate as "**|accuracy_MLC −
accuracy_host| ≤ 2 percentage points**, and both accuracies are ≥ 90%
absolute." That is each pipeline's accuracy vs ground truth (which under
v4 is by-file), NOT host-vs-silicon agreement.

`compare_decisions.py` reports "FAIL" because it checks bit-identity of
class labels across all 3396 windows (44 disagreements → "FAIL: 98.704%
agreement"). That's a strict-equality diagnostic, useful for first-
divergence reporting, but it is NOT the §9 gate. The gate is the
quantitative threshold computation in the table above. Worth being
explicit about this in the paper writeup — a reviewer reading just
compare_decisions's "FAIL" output might think the gate fails.

### Disagreement structure: 40 motion-arm windows + 4 still-arm windows

Host and silicon agree on 3352 of 3396 windows. The 44 disagreements
break down as:

| Arm | Type | Count |
|---|---|---|
| Still | host class=4, silicon class=0 (host false-positive) | 4 |
| Motion | host class=0, silicon class=4 (host false-negative) | 39 |
| Motion | host class=4, silicon class=0 (host false-positive) | 1 |

Pattern in disagreements:
- **All 40 motion-arm host-class=0 windows have host_p2p_norm in
  [0.031, 0.049]**, all just below the trained threshold 0.049316.
- Silicon presumably computes p2p_norm slightly above threshold on
  these windows (silicon's actual p2p values are not exposed; AN5259
  §1.3 confirms internal-only).
- **Implication**: host's p2p is consistently ~0.001–0.005 g lower than
  silicon's on borderline-amplitude windows.
- Disagreement timestamps don't cluster at the 1-second servo period;
  they cluster at multiples of ~12 seconds — the beat between host's
  ~0.706 s window cadence and the 1-second servo period. This is
  phase-aliasing of window boundaries against servo turnarounds, not a
  direct servo-event coupling.

Window 0 on the still arm is an outlier worth noting separately:
host_p2p_norm = 0.93 (18× threshold), host claims class=4, silicon
claims class=0. **This is the IIR1 HP filter's startup transient.**
The filter state initializes at zero; on the first sample (gravity-
dominated ~−0.97 g on Z), the filter output spikes for the first
~25-30 samples before settling. Window 0 covers the first 75 MLC
samples (= 150 sensor samples = ~0.72 s) and contains the entire
transient. Silicon presumably has the same filter init, but
half-precision math may attenuate the transient differently. This is
ONE of the 4 still-arm errors; the other 3 are real low-amplitude
events similar in character to the 40 motion-arm cases.

### Hypothesis 1 (FP16 emulation): FALSIFIED

Wrote `fp16_emulate.py`, a faithful Python reimplementation of
`pc_step` with a precision toggle. Verified fp32 mode against
replay_parity to within float32 ULP on all 1698 motion-arm windows
(bit-identical p2p values, bit-identical class labels, bit-identical
row count).

Hypothesis: silicon's MLC computes the IIR1 filter and L2 norm in
half-precision (FP16) arithmetic per AN5259 §1.2; host's float32
implementation accumulates slightly different values over a 75-sample
window, biasing host's p2p_norm 0.001–0.005 g lower than silicon's at
borderline amplitudes.

Test: re-run with `--mode fp16` (every filter intermediate cast to
numpy.float16). Result on the 40 motion disagreement windows:

| Outcome | Count |
|---|---|
| FP16 recovered the disagreement (now class=4, matching silicon) | **0** |
| FP16 still disagrees (still class=0) | 39 |
| Net new disagreement introduced by FP16 (was class=4, now class=0) | 1 |

FP16 p2p deltas vs fp32 are typically 0.0001–0.0005 g — far smaller
than the ~0.005 g gap needed to cross threshold. Deltas are not
systematically biased toward silicon's reported behavior; they're
roughly symmetric noise around fp32 values.

**Conclusion**: the disagreement mechanism is NOT aggressive end-to-end
FP16 arithmetic. AN5259 only commits to FP16 for stored *coefficients*,
not arithmetic intermediates, and this experiment is consistent with
silicon doing the actual arithmetic at a precision higher than FP16.

This is a clean negative result: a plausible mechanism is ruled out
with code that's in the repo (`code/analysis/fp16_emulate.py`) and
output artifacts that can be regenerated and audited.

### Hypothesis 2 (bus-contention sample gaps): FALSIFIED

Wrote `accel_gap_audit.py`. Hypothesis: imu_logger's flock-protected
I²C bus access blocks while mlc_poller holds the bus (for MLC0_SRC
bank-switch reads). During blocking periods, imu_logger's per-sample
timing skews, producing irregular timestamps in accel.csv. The host
replay pipeline (which assumes uniformly-spaced 208 Hz samples) sees
slightly different data than silicon's internal sample stream,
explaining the ~0.005 g p2p discrepancy at borderline windows.

Test: compared sample-to-sample timestamp distributions in the 40
disagreement windows against a control set of 40 agreement windows
(every k-th window not in the disagreement set, k = n_windows / 40).

| Metric | Disagreement windows | Control windows |
|---|---|---|
| mean of per-window max_gap | 5.03 ms | 5.57 ms |
| max of per-window max_gap | 5.23 ms | **26.77 ms** |
| per-window mean_gap | 4.7086 ms | 4.7086 ms |
| total gaps > 3× expected (14.4 ms) | **0** | 1 |
| total gaps > 2× expected (9.6 ms) | **0** | 1 |

Expected interval = 1/208 Hz = 4.808 ms. Sensor-side actual interval =
4.7086 ms (consistent with the ~212 Hz oscillator drift documented on
2026-05-22). Disagreement-window max-gaps are all 4.94–5.23 ms — within
8% of expected. The single >3× gap-outlier in the 80-window sample
falls in a control window (1260, max=26.77 ms, std=1.99 ms — likely a
single delayed sample), not a disagreement window.

**Conclusion**: bus contention does not produce detectable
sample-timing irregularities at the disagreement windows. The host
replay is operating on a uniformly-sampled accel.csv. The disagreement
mechanism is not sample-gap related.

### Where this leaves us

Two plausible precision-/timing-related mechanisms tested and falsified.
The remaining ~1.05 pp gap between host and silicon aggregate accuracy
must originate in some other implementation difference not specified by
AN5259. Candidates I have NOT investigated tonight (any of which is a
reasonable next step, though not required for the gate):

1. Silicon's L2 norm formula. AN5259 doesn't specify whether silicon
   computes √(x²+y²+z²) directly or uses a fast-inverse-square-root
   approximation, a Newton iteration, or a lookup table. parity_core
   uses libm sqrtf. A difference of 0.001 in the norm propagates
   through the recursive IIR1 filter and could plausibly produce the
   observed 0.005 g p2p bias.
2. Silicon's IIR1 input may be (norm − DC_estimate) rather than norm
   directly. HP-filtering a gravity-dominated signal benefits from
   subtracting the ~1.0 g DC level first. parity_core feeds raw norm
   to the filter.
3. Silicon's 2:1 decimation phase may be hardware-anchored to a specific
   sample (not necessarily phase 0 as parity_core assumes), AND/OR
   silicon may apply an anti-aliasing low-pass before decimation that
   AN5259 doesn't expose.

None of these is testable without additional information from ST or
silicon-side instrumentation that we don't have. The honest paper
sentence: "**The §9 gate passes at 1.05 pp; the remaining gap between
host and silicon corresponds to host computing systematically slightly
lower peak-to-peak values on borderline-amplitude windows. Two precision-
and timing-related mechanisms (full FP16 arithmetic emulation and
imu_logger bus-contention sample-skew) were tested and falsified. The
precise silicon-side computational pathway is not fully specified by
AN5259 §1.2 and is treated as unknown for the purposes of this work.**"

### Implication for v4 amendment "implementation defaults"

v4 explicitly authorizes tuning three implementation defaults — variance
estimator (biased/unbiased), threshold operator (lte/lt), and IIR filter
sign convention (A/B) — based on parity gate results. Reading the code
and tree.json tonight:

- **Variance estimator** is "biased" (1/N). Doesn't matter for this
  result because the tree splits on p2p, not variance.
- **Threshold operator** is "lte". Doesn't matter because no
  disagreement windows have host_p2p exactly at 0.049316; all are
  bounded below by ~0.005 g.
- **Filter sign convention**: parity_core uses Convention A
  (y = b1·x + b2·x⁻¹ − a2·y⁻¹); `mlc_json_to_parity.py` sign-flips a2
  when extracting from MEMS Studio's Convention B. Verified
  mathematically: MEMS Studio's a2 = +0.941339 corresponds to
  parity_core's a2 = −0.941339, which is what tree.json contains. The
  sign-flip was applied correctly. Filter math is the same equation as
  silicon would compute, modulo arithmetic precision.

**No tuning of v4's three knobs would change the outcome.** The
gate-passing result stands as-is; the unresolved disagreement mechanism
is in some part of silicon's computation that is not parameterized in
parity_core because AN5259 doesn't specify it.
