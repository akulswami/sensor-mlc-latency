## PCA9685 setup on bus 1

Goal: bring up SG90 servo control via PCA9685 on a separate I2C bus from
the sensor, in preparation for training data collection.

### What happened

1. Verified bus 1 is exposed on Jetson 40-pin pins 27/28 by temporarily
   moving the sensor wires there and confirming 0x6A appeared on bus 1.
   Restored sensor to pins 3/5 afterward.
2. Identified address conflict risk: PCA9685 ships at default 0x40,
   colliding with INA3221 (Orin Nano carrier power monitor) at the same
   address. Confirmed empirically: with PCA9685 connected at default
   0x40, tegrastats reported VDD_IN 0mW/0mW (INA3221 broken by
   contention). With PCA9685 disconnected, tegrastats reported expected
   ~4500 mW idle power.
3. Soldered a bridge across what was intended to be the A0 address pad
   to move the chip to 0x41. Continuity verified with multimeter.
4. Reconnected PCA9685 to bus 1. i2cdetect showed the chip at 0x60,
   not 0x41 — meaning the pad bridged was actually A5, not A0. The
   physical pad layout on the HUAREW clone board is hard to read
   visually; the wrong pad was bridged.
5. Accepted 0x60 as the working address rather than desoldering and
   retrying. 0x60 is clear of all other devices on bus 1 (0x25, 0x40),
   clear of the PCA9685 all-call (0x70), and tegrastats confirms
   INA3221 is healthy (VDD_IN ~6500 mW).

## PCA9685 functional test

Configured chip for nominal 50 Hz PWM (PRESCALE=0x79=121) with 7.5%
duty cycle on channel 0 (commanded 1.5 ms pulse intended as servo
neutral). Verified output on Saleae D2 probing channel 0 PWM pin.

### Measured

- Frequency: 55.876 Hz (vs. 50 Hz commanded)
- Duty cycle: 7.5% (matches command exactly)
- Period: 17.9 ms

### Interpretation

Duty-cycle accuracy is exact (tick-counting is precise), but the
internal 25 MHz oscillator runs ~12% high. Per NXP, the internal
oscillator is untrimmed with documented ±20% chip-to-chip variation
(community forum: https://community.nxp.com/t5/Other-NXP-Products/PCA9685-drive-frequency-tolerance-details/m-p/1732787).
This chip's behavior is within spec.

### Decision

Accept the measured clock; do not calibrate. Implications for this
experiment:

- Commanded pulse widths are interpreted relative to this chip's
  actual clock, not nominal 25 MHz. A "1.5 ms" command produces a
  ~1.34 ms actual pulse (17.9 ms period × 7.5%).
- The training-data-spec already specifies PWM endpoints as
  measured values reported with uncertainty (not as setpoints).
- Saleae captures the actual PWM signal as ground-truth label,
  so labeling accuracy is unaffected by clock error.
- Servo angle endpoints will be set empirically once the rig is
  built — whatever pulse widths produce the desired 0°↔150°
  motion become the operational parameters.

## Servo functional test

Connected SG90 to PCA9685 channel 0. External 5V supply (USB-A power
brick + breakout cable) wired to V+ via the middle pin of an unused
output channel header (avoided the screw terminal after the first
PCA9685's terminal was unreliable).

### Failure mode encountered

Initial wiring: USB breakout red wire mistakenly connected to VCC
(logic supply, 3.3V from Jetson) instead of V+ (servo rail). This
back-fed the Jetson 3.3V rail with 5V from the USB brick. Caught
before extended damage; tegrastats remained healthy after unplugging.

After correcting the wiring (USB +5V to V+, not VCC), Saleae A3
showed 5.52V at the servo's V+ pin and PWM signal on D2 was clean.

### Servo did not visibly move on first power-up

Cause: commanded 1.5 ms pulse (servo neutral nominal) corresponded
to a position the horn was already sitting at, so no visible motion
occurred. Verified servo is alive by commanding 0.6 ms pulse
(`i2cset -y 1 0x60 0x08 0x7B; i2cset -y 1 0x60 0x09 0x00`) which
produced visible motion to one extreme. Servo returned to center
with the 1.5 ms command.
