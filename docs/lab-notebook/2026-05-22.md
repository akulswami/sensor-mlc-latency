length 75 (one of {25, 75, 200} candidates), MLC ODR 104 Hz, accel

### MEMS Studio 2.3.1 quirks learned (documented in README)

1. **Convention B for IIR filter coefficients.** UI sign for a2 is
   opposite of AN5259. Verified empirically via scipy.signal.freqz:
   under MEMS Studio's convention, a2=+0.941339 with b1=0.970669,
   b2=-0.970669 gives HP at fc=1 Hz on a 104 Hz sample rate. The
   tree.json extractor sign-flips a2 when emitting parity_core's
   schema (parity_core implements AN5259 convention).
2. **features.arff uses CRLF line endings.** Tooling handles.
3. **In-memory class list staleness.** First ARFF generation
   failed with "classes don't match configuration" until MEMS
   Studio restart forced reload from disk.
4. **"Percentage of data to use for training" actual behavior
   unclear.** Reported accuracy is over all samples (Instances=6734
   = full corpus), suggesting no real holdout. Doesn't matter for
   us; we use session 3 as the real holdout.

### Note on the flicker at probe ms 18978-19685

During motion phase 3, MLC briefly read 0x00 (still) for ~700 ms
before recovering to 0x04. One window's worth at 75-sample/104-Hz
cadence is ~721 ms, so this is exactly one MLC window where
p2p_norm dropped below 0.049316.

Most plausible cause: at servo turnaround (TO_MAX and TO_MIN
endpoints), velocity passes through zero. If a 721-ms MLC window
straddles two consecutive turnarounds without much in-between
motion, the p2p_norm of that window can dip below the trained
threshold. Servo period_ms is 1000, so adjacent turnarounds are
~500 ms apart — MLC windows ARE longer than the servo half-cycle,
which means the window can sometimes catch two turnaround events
where the signal briefly resembles "still".

Not investigating tonight. Two reasons:
(a) The flicker is in-distribution training-noise behavior, not
    broken silicon.
(b) If this is real, BOTH the host classifier and the silicon
    classifier should produce the same flicker when run on the
    same accelerometer stream. The parity gate is unaffected by
    classifier "weirdness" — only by host-silicon disagreement on
    identical input.

### Note: rejected proposal to retune the servo

I proposed (mid-session) speeding up the servo to avoid the
flicker. Claude pushed back hard, correctly:

- The trained threshold (0.049316) was learned from data at the
  original servo speed. Faster motion produces larger p2p values
  trivially above threshold — the classifier will look perfect on
  faster motion not because it's better but because we moved out
  of the borderline regime where errors actually happen.
- Session 3 (held-out test data) was collected at the original
  speed. Changing the silicon-side speed breaks the
  host-vs-silicon comparison on equivalent stimulus.
- The borderline regime is actually MORE interesting for the
  paper. Showing host-side beats sensor-side at the EASY case
  (no errors) is unsurprising. Showing it at the HARD case
  (borderline motion, real misclassifications, latency differences
  matter) is the actually-interesting headline.

Decision: keep the servo program unchanged for the parity gate.
Treat the flicker as data about the classifier, not a bug.
