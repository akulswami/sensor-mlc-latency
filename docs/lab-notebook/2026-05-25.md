## Orchestrator dispatch bug — root cause and fix

The bug was in `code/orchestrator/run_session_parity.py` line 73:

```python
JETSON_MLC_SETUP = f"{JETSON_REPO}/code/jetson/session4/mlc_setup"
```

A hardcoded binary path. The `--mlc-config-header` argument was
stored in `session.json` for provenance but never used to select
which binary to flash. Every prior session that "ran w=25" or
"ran w=200" actually ran the same `mlc_setup` binary, which had
been compiled from `mlc_motion_w75.h`.

The misattribution was invisible at every layer above the
orchestrator: `session.json` recorded the header argument, the
lab notebook recorded the session as "w=25" or "w=200", `§9`
evaluations were performed against the corresponding trees. The
only ground truth that would have caught it — actual silicon
behavior — was hidden behind the polling layer (no one ever
read silicon's window-length register).

This affects every session in the repo from S4-prime onward
(S4-prime, S5, S6, S7, S7-prime). All of them are w=75 silicon
regardless of what `session.json` records. The §9 PASS result
yesterday (S7-prime, 0.47 pp gap) is real and remains valid
*for w=75*; the §9 FAIL result yesterday at "w=25" was an
artifact of running the w=25 tree analysis against w=75 silicon
classifications.

### Fix: Strategy A (dispatch by header)

Replaced the hardcoded constant with a function that derives
the binary path from `--mlc-config-header`:

```python
def jetson_mlc_setup_path(mlc_config_header):
    m = re.match(r"^mlc_motion_w(\d+)\.h$", mlc_config_header)
    if not m:
        raise ValueError(...)
    window_n = m.group(1)
    return f"{JETSON_REPO}/code/jetson/session4/mlc_setup_w{window_n}"
```

Threaded `mlc_setup_path` through `verify_jetson_state_parity()`,
`run_mlc_setup()`, and `run_class_capture_parity()`. Made
`--mlc-config-header` required (no default) so the user must
consciously pick a window length each time.

Built `mlc_setup_w75` on the Jetson tonight (the other binaries
already existed but were unused). The sha256 of the new
`mlc_setup_w75` matches the unsuffixed `mlc_setup` exactly
(`59f9b3c0...`), empirically confirming the historical
`mlc_setup` was always the w=75 build.

### Strategy C': silicon liveness check after flash

Added `verify_silicon_alive()` that, after `mlc_setup` reports
"MLC ready", reads MLC0_SRC over SSH via i2cset/i2cget. The
three-step bank-switch sequence (FUNC_CFG_ACCESS = 0x80, read
MLC0_SRC = 0x70 in embedded bank, restore FUNC_CFG_ACCESS =
0x00) is the same protocol used in `mlc_poll_probe.c` for
runtime silicon polling. Asserts MLC0_SRC is in {0x00 still,
0x04 motion}; any other value (0xFF garbage, unparseable
response, i2c error) raises.

**The check is a liveness check, not a configuration check.**
It verifies silicon is responsive and classifying. It does NOT
verify which window length is deployed; that's the post-hoc
diagnostic's job. The trade-off was conscious: a true
window-length readback would require reading from the MLC's
internal page register at address 0xF2, which goes through a
different (and undocumented in our repo) page-protocol that I
could not reverse-engineer tonight without the AN5259
datasheet. Strategy C' as implemented is sufficient for
catching "silicon didn't get configured" failures; the
post-hoc transition-density diagnostic catches the rest.

### Strategy C' caveats worth recording

The liveness check is brittle to silicon's transient state
immediately after flash. The first ~85 ms after `mlc_setup`
returns, MLC0_SRC may oscillate between 0x00 and 0x04 as the
IIR filter settles. Both values are accepted by the liveness
check, so the check passes harmlessly even in this transient
window. But the printed message ("Silicon liveness OK:
MLC0_SRC = 0x04 (motion)") can read alarmingly on a still rig.
Tonight's w=25 capture had `MLC0_SRC = 0x04` reported at flash
time for both arms, and both arms produced correct
classifications throughout. The printed value should not be
interpreted as "the rig is in motion right now."

## The methodological finding: silicon's MLC0_SRC register-holding

Initial framing of the §9 result was "host pipeline has a bug
at short windows." Investigation found that framing is wrong.

The host classifier is correctly computing peak-to-peak over
its 240 ms windows. At servo direction reversals — the moment
the servo command flips MIN→MAX or MAX→MIN — the servo
mechanically pauses for ~200 ms before accelerating in the new
direction. During that ~200 ms pause, the accelerometer is
genuinely stationary. Host's window covering that pause sees
genuinely-stationary accel data: A_X mean = -0.0547 g, std =
0.0013 g (51 samples), peak-to-peak = 0.010 g. That's below the
threshold (0.029 g). Host correctly classifies it as class-0.

But silicon reports class-4 throughout that interval. Why?
Because silicon's MLC engine holds the previous window's
classification in MLC0_SRC until the next window completes.
The previous window — which DID see real motion — set MLC0_SRC
= 4. The current window covering the pause hasn't completed
yet, so the register hasn't updated.

The 12 silicon polls during host's window 5 (1.17-1.41s after
motion arm start) all read class-4. The accel data over the
same interval shows the rig genuinely at rest. Both classifiers
are correct about their own data slice; silicon's register is
just temporally lagging.

### Implications for §9

`silicon_align.py` uses the rule "the silicon class for host
window N is the last silicon poll falling inside window N's
time range." This rule has an unstated assumption: that the
silicon poll's value reflects silicon's classification of the
same data host window N saw. The assumption holds when
silicon's window boundaries are time-aligned with host's, but
fails when they're not.

At w=75 (720 ms windows), silicon's window completions happen
infrequently and silicon's register updates dominate the poll
stream. The temporal alignment artifact exists but is small,
and the w=75 §9 evaluation shows it: 0.47 pp gap on the
combined accuracy.

At w=25 (240 ms windows), silicon's window completions are
frequent but the held-register intervals between them are
roughly equal to or larger than host's window length. The
temporal alignment artifact is significant: host registers
mid-burst motion-free intervals, silicon's register holds the
previous burst's classification, the alignment rule treats both
as classifications of the same data slice, and they disagree.

### Implications for the paper

This is not a sensor-system *limitation* — it's a sensor-system
*characteristic* worth knowing about. The IEEE Sensors Letters
paper has space for ~2-3 sentences on it in the discussion
section, e.g.:

> The on-sensor MLC reports decisions through a register (MLC0_SRC)
> that holds its value between window completions. When sampling
> this register at sub-window cadence, the reported classification
> reflects the most recently completed window's decision rather
> than the instantaneous sensor state. This characteristic affects
> the interpretation of classifier-output traces but does not
> affect the latency measurement, which is keyed to the rising
> edge of fresh decisions.

The headline finding for the paper — that on-host inference at
208 Hz can be lower-latency than the on-sensor MLC's
~12.5 ms hardware floor — is unaffected by register-holding.
Latency is measured at the moment of a fresh decision, not from
held register states.

### Implications for §9 going forward

§9 as currently formulated has known temporal-aliasing
limitations at short window lengths. Three possible
reformulations exist:

1. **Acknowledge and document**: §9 results at short windows
   include known temporal-aliasing artifacts that don't reflect
   genuine classifier disagreement. Use w=75 (where the artifact
   is small) as the primary parity gate; record short-window
   results with this caveat.

2. **Phase-align host's windows to silicon's**: change
   `replay_parity` to start its first window at the same
   `mlc_sample_count` silicon's first window started at.
   Requires knowing silicon's window-boundary phase, which is
   not directly observable through MLC0_SRC.

3. **Define ground truth from sweep.log and evaluate host vs
   silicon vs ground truth separately**: a more elaborate
   reformulation that abandons the host-vs-silicon comparison
   in favor of independent ground-truth evaluation.

(1) is sufficient for the IEEE Sensors Letters paper's
purposes. (2) and (3) are future work, not blockers.

## Methodological lessons

Two come out of tonight cleanly:

1. **Binary identity is not configuration identity.** Tonight's
   investigation of "is `mlc_setup_w25` actually flashing w=25?"
   went through three confirmations: sha256 differs from
   `mlc_setup_w75` (yes), behavioral test on still rig is clean
   (100/100 class-0), motion arm shows expected transition rate.
   The MLC config header writes 93 register bytes; only a handful
   differ between w=25 and w=75. A binary's sha256 differing from
   another's is necessary but not sufficient evidence that they
   flash different silicon states. Behavioral verification is the
   ground truth.

2. **Holding registers create temporal aliases that look like
   bugs.** The §9 result that looked like "host has a bug at
   short windows" was actually "silicon's register-holding
   creates artifactual agreement at long windows that breaks
   down at short windows." This is a generalizable failure mode
   for any host-vs-on-chip comparison where the on-chip output
   is exposed through a held register. Worth filing for the
   paper's discussion.

## Btest campaign

The 59-block btest campaign produced the empirical basis for the
v7.5 amendment. Key findings at btest scale (n=15-18 per cell,
vanilla scheduling):

- **host idle median: 351 µs** (n=18)
- **mlc bank-switch idle median: 761 µs** (n=17)
- Direction is opposite of pre-reg H1 — host is FASTER than MLC.
- mlc-binary idle median: 274 µs (n=18), with high bimodal sd
  due to gpiod jitter
- Under i2c-contention: host 643 µs, mlc 1534 µs

Mechanism: `read_mlc_src()` performs 3 sequential I²C transactions
per measurement (write bank-select, read MLC0_SRC, restore bank).
Predicted latency = 3 × i2c_read_bench, which matches observed
within ~20% at both idle and contention. The MLC silicon's
inference time is not the bottleneck; the bank-switch read
protocol is.

CPU stress (the originally pre-registered stress modality) was
checked at btest and produced approximately no median latency
change. I²C contention (the new stress modality, calibrated to
N=3 i2c_hammer processes) produced 290-770 µs latency increases.
The pre-registered stress modality was the wrong choice; the
new one captures the actual bottleneck.

## cpufreq investigation

To understand the energy reversal, I dumped CPU freq state and
discovered:

- `scaling_min_freq = 729600` for all CPUs (not 1728000)
- `scaling_max_freq = 1728000` for all CPUs (jetson_clocks set
  this)
- `scaling_cur_freq` varying between 729600 and 1728000

This meant jetson_clocks's pin was being defeated. Investigating
further: `sudo nvpmodel -q --verbose` showed that the active
power mode ("25W", which is mode 1, the system default) declares
`CPU_A78_0: MIN_FREQ = 729600`. Whenever nvpmodel reasserts its
configured state, `scaling_min_freq` returns to 729600,
defeating jetson_clocks.

Crucially: the tegrastats log records CPU frequencies. Reading
them showed:

- **b700 mlc i2c-contention**: all 6 CPUs at 1728 MHz throughout
  (jetson_clocks WAS effective during this block).
- **b701 host idle**: CPUs at MIXED freqs (729 MHz and 1728 MHz)
  throughout (jetson_clocks was NOT effective during this block).
- **b702 mlc idle**: all 6 CPUs at 1728 MHz throughout (jetson_clocks
  WAS effective, because my re-apply at 23:34 stuck through the
  full block).

This explains the apparent reversal: b701 and b702 were measured
under different power profiles. The 1951 mW gap is the cost of
jetson_clocks (1.7-2 GHz idle CPU pinning), not the cost of
running MLC. **The "MLC saves energy at idle" comparison is
not testable from b701 vs b702** because the underlying clock
states differ.

## Methodological lessons recorded

1. **jetson_clocks effectiveness must be verified empirically
   from tegrastats CPU-freq fields, not assumed from
   `jetson_clocks --show` immediately after apply.** nvpmodel
   can reassert MIN_FREQ at any time, and the timing is
   non-deterministic.

2. **Per-block jc_eff (% of tegrastats CPU-freq samples at ≥
   1700 MHz) is a clean post-hoc test for jc effectiveness.**
   This metric should be computed and stored in
   block_metadata.json for every measurement going forward.

3. **The btest "+155 mW MLC saves" finding does NOT apply at the
   long-duration scale**, because btest was measured under
   free-running DVFS (jc was never applied; or briefly applied
   and immediately defeated). At long-duration scales with jc
   actively effective, the energy gap is expected to collapse
   to near-zero (apples-to-apples test pending b703).

4. **The exclusion-rate problem from §11 is real at scale.** At
   n=18 (btest) the cell-level exclusion rate was 0%. At
   n=360 (smoke) it was 12-17%. The classifier oscillation that
   v7.4 documented as a known risk is materializing at large n.
   The §11 10% cap will be exceeded by the confirmatory campaign
   as currently planned.
